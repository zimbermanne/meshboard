# MeshBoard — Technical Specification
## For AI-assisted development

---

## 1. Product overview

MeshBoard is a local-first, offline-capable advertising broadcast network for Android.
It allows individuals and small businesses to broadcast short text advertisements
to all nearby devices — with or without internet — using a combination of
Bluetooth, WiFi Direct, and standard WiFi/internet.

Think of it as a digital community notice board that travels on people's phones.
Messages are public broadcasts, not private chats. Every message expires after
a set duration. Users pay to post. The network runs itself through human movement.

Target market: Small traders, vendors, and businesses in low-connectivity areas.
Initial deployment: Arusha, Tanzania.

---

## 2. Two separate applications

### A. Client Node App (Android)
- Downloaded by all users
- Works fully offline
- Receives and displays broadcasts
- Sends post requests to super-node
- Syncs with nearby nodes automatically in background
- Text + emoji only — no images, no video, no audio

### B. Super-Node Admin App / Dashboard (Android or web)
- Operated by the network owner only
- Runs on dedicated always-on hardware (Raspberry Pi or similar)
- Requires internet connection for payment top-ups only
- Authorizes all post requests before broadcast
- Manages user credits and token generation
- Acts as network anchor and archive

## 3. System architecture

### Communication layers (in priority order)
The app automatically detects and uses the best available connection:

1. Internet (fastest, longest range)
2. Local WiFi / LAN (medium range, no internet needed)
3. WiFi Direct / P2P (device to device, ~50-100m)
4. Bluetooth Low Energy (last resort, ~10m)

The user never manually selects a layer. The app handles this silently.

### Network topology
- Every phone is a node
- Every node carries all active broadcast messages (as encrypted blobs)
- When two nodes meet (via Bluetooth or WiFi Direct), they sync automatically
- The super-node is an always-on anchor node with higher authority
- There is no central server — the network IS the phones
- This is a gossip protocol / epidemic network architecture

### Sync is bidirectional
Every sync session between any two nodes (including super-node) goes both ways:

Node → Super-node pushes:
- New device registration (ID + display name)
- Post requests awaiting approval
- Token redemption requests
- Acknowledgement of received messages

Super-node → Node pushes:
- Approved broadcast messages
- Credit balance updates
- Token validation responses
- Expired message deletions (cleanup list)

### Sync through relay nodes
A new user does not need to physically reach the super-node.
If Node A meets Node B, and Node B recently synced with the super-node,
Node B can carry Node A's registration to the super-node on the next sync.
The super-node learns about new users through the movement of existing users.

### Sync priority order
1. Token redemptions and credit updates (money — always first)
2. New registrations
3. Approved broadcasts and expirations
4. Display name updates and network statistics

### Conflict resolution
Every data change is timestamped on the originating device.
When two nodes have conflicting versions of the same record,
the most recent timestamp wins. No manual resolution required.

## 4. Identity system

### System ID
- Generated by the app on first install
- Format: NODE-XXXX-XXXX (e.g. NODE-7X92-KM44)
- Generated using UUID v4, formatted for readability
- Permanent — never changes, survives phone replacement
- Generated fully offline — no server required
- This ID is the user's permanent identity in the network
- All tokens and payments are tied to this ID
- If a user gets a new phone, they enter their old ID manually to restore

### Display name
- Chosen freely by the user (e.g. "Mama Pima Boutique")
- Can be changed at any time
- This is what other users see in the broadcast feed
- Not unique — two users could have the same display name
- The system ID is the true unique identifier, not the display name

### Registration flow
1. User installs app
2. App generates NODE ID instantly (offline)
3. User enters their chosen display name
4. App begins syncing with any nearby node
5. Registration data propagates to super-node via relay nodes
6. No email, no password, no office visit, no verification required

## 5. Broadcast system

### What a broadcast contains
- Sender display name
- Message text (max 280 characters, text and emoji only)
- Optional: one website/social media link
- Optional: one phone number
- Package duration (days)
- Broadcast timestamp (set by super-node at authorization)
- Expiry duration in seconds (calculated from package)
- Unique message ID
- Sender NODE ID

### Message flow
1. User composes message in client app
2. User selects package duration (free or paid)
3. App sends post request to super-node (directly or via relay)
4. Super-node operator reviews in approval queue
5. Operator approves or rejects
6. If approved: super-node timestamps the message and broadcasts to all connected nodes
7. Each node that receives the message starts its own local countdown immediately
8. Message displays in feed until local countdown reaches zero
9. On countdown zero: message auto-deletes from that device
10. Message is never retroactively removed from devices that already received it

### Expiry rule — critical
The countdown starts the moment a node RECEIVES the message,
not when it was originally sent or approved.
This means:
- A node that was offline for 3 days and then receives a 7-day post
  will display it for the full 7 days from the moment of receipt
- There are no missed countdowns
- Late-syncing nodes always get the full advertised duration
- This is a feature, not a bug — advertised as guaranteed full duration delivery

### Expired message handling
- Super-node sends a cleanup list to nodes: message IDs that have expired globally
- Nodes delete those messages on next sync
- Messages past their maximum possible age are never synced to new nodes

## 6. Free post and paid packages

### Free monthly post
- Every registered user gets 1 free broadcast per calendar month
- Duration: 2 days
- Must still go through super-node approval queue
- Free post resets on the 1st of each month
- Tracked per NODE ID on the super-node

### Paid packages
| Duration | Price |
|----------|-------|
| 1 day    | $1.00 |
| 2 days   | $2.00 |
| 3 days   | $3.00 |
| 4 days   | $4.00 |
| 5 days   | $5.00 |
| 6 days   | $6.00 |
| 7 days   | $7.00 |

Payment is via prepaid credits loaded onto the user's account.
Credits are deducted at the time of post approval, not at submission.

## 7. Credit and token system

### How credits work
- Credits are stored on the super-node, linked to a NODE ID
- Credits are purchased in advance (prepaid)
- When a post is approved, the corresponding credit amount is deducted
- Credit balance is synced back to the client node after every transaction
- The client app shows the user their current balance

### Token generation (operator side)
When a user pays cash or M-Pesa to the operator:
1. Operator opens super-node dashboard
2. Enters NODE ID (or searches by display name) and amount paid
3. System generates a unique one-time token
   Format: TXN-XXXX-XXXX-XXXX (e.g. TXN-8821-NODE-7X92)
4. Token is stored with:
   - Token ID
   - Tied to NODE ID (specific user only)
   - Credit amount
   - Created timestamp
   - Expiry timestamp (48 hours after creation)
   - Status: Pending
   - Created by: operator ID
5. Operator communicates token to user (verbally, SMS, or written)

### Token redemption (user side)
1. User opens client app → Credits screen → Redeem token
2. User enters token string
3. App sends: token + NODE ID to super-node
4. Super-node validates:
   - Token exists
   - Token status is Pending (not already redeemed or expired)
   - Token is tied to this specific NODE ID
   - Token has not expired (within 48 hours)
5. All checks pass → credits added to NODE ID account → token marked Redeemed
6. Any check fails → rejection message shown, no credits added
7. User receives in-app receipt showing amount, token ID, and timestamp

### Token security
- Tokens are single-use only
- Tokens are tied to one specific NODE ID — cannot be used by anyone else
- Tokens expire after 48 hours if unredeemed
- All tokens are permanently logged — created, redeemed, and expired records kept forever
- This log enables recovery: if a user loses their token, operator can look up
  all pending tokens for that NODE ID and re-communicate the token string

### Token recovery
1. User reports lost token to operator
2. Operator searches dashboard by NODE ID or display name
3. Operator sees full token history for that user
4. Operator reads out any Pending token to user
5. User redeems normally
No new token needed. No money lost.

## 8. Payment flow

### V1 payment methods
- Cash paid directly to super-node operator
- M-Pesa paid to operator's registered number (operator confirms manually)

### V2 future automation (not in scope for v1)
- M-Pesa Daraja API integration
- On payment received: auto-generate token, auto-send via SMS to user
- No operator intervention needed

### Payment record structure
Every payment logged permanently on super-node:
- Payment ID
- NODE ID of payer
- Amount
- Method (cash / M-Pesa)
- Operator who confirmed it
- Token generated from this payment
- Timestamp

## 9. Data storage

### Super-node stores (permanent archive)
- All NODE registrations (ID, display name, registration date)
- All credit balances per NODE ID
- All token records (full history)
- All payment records
- All post requests (approved, rejected, pending)
- All broadcast history
- Network statistics

### Client node stores (operational only)
- Own NODE ID and display name
- Own credit balance
- Active broadcast messages (with local countdown timers)
- Own post history
- Last sync timestamp
- Encrypted message blobs destined for other nodes (carried temporarily)
- Cleanup list of expired message IDs

## 10. Client node app — screens

### Screen 1: Feed (home)
- List of all active broadcasts sorted by newest first
- Each card shows: display name, message text, optional link, optional phone number
- Countdown bar on each card showing time remaining
- Bar color: green (plenty of time) → amber (less than 12 hours) → red (under 1 hour)
- Auto-removes cards when countdown reaches zero
- Shows number of nodes currently in range

### Screen 2: Compose / Post
- Text input (max 280 chars, text and emoji only)
- Optional link field
- Optional phone number field
- Package selector: Free (if available), $1 through $7
- Shows current credit balance
- Shows free monthly post availability
- Submit sends post request to super-node for approval
- User sees pending status until approved or rejected

### Screen 3: Credits
- Current credit balance
- Free monthly post status (available / used, resets on date X)
- Package list with prices
- Redeem token input field
- Payment instructions (visit operator or M-Pesa number)

### Screen 4: Profile
- Display name (editable)
- NODE ID (visible, read-only — user should note this down)
- Member since date
- Total posts sent
- Active posts count
- Post history list

### Screen 5: Network status
- Connection mode indicator (internet / WiFi / Bluetooth)
- Nearby nodes count
- Super-node connection status
- Last sync timestamp
- Number of messages being carried for other nodes
- Manual sync button

## 11. Super-node dashboard — screens

### Overview (home)
- Total nodes registered
- Active broadcasts count
- Monthly revenue
- Pending approval count (with alert if > 0)
- Last sync timestamp
- Online / offline status indicator

### Approval queue
- List of all pending post requests
- Each shows: NODE ID, display name, message text, link, phone, package selected, credit cost
- Approve button → deducts credits, timestamps, broadcasts to network
- Reject button → notifies user, credits not deducted
- Operator can add a rejection reason (optional)

### Node registry
- Searchable list of all registered nodes
- Search by NODE ID or display name
- Each node record shows: ID, display name, registration date, credit balance,
  total spent, posts sent, active posts, full token history, full payment history

### Token management
- Generate new token: search user by NODE ID or display name, enter amount, generate
- Shows generated token string for operator to communicate to user
- Token history: all tokens with status (Pending / Redeemed / Expired)
- Filter by user, status, or date

### Payment log
- Full payment history across all users
- Filter by date, user, method (cash / M-Pesa)
- Revenue totals by day, week, month
- Revenue breakdown by package duration

### Active broadcasts
- All currently live broadcasts with countdown bars
- Operator can manually expire a broadcast if needed (moderation)
- Filter by user or time remaining

## 12. Technology recommendations

### Client node app
- Language: Kotlin (Android native)
- Minimum Android version: Android 8.0 (API 26) — covers ~95% of Tanzanian Android devices
- Bluetooth: Android BLE (Bluetooth Low Energy) API
- WiFi Direct: Android WiFi P2P API
- Local database: Room (SQLite wrapper for Android)
- Background sync: WorkManager (Android background tasks)
- Encryption: AES-256 for stored message blobs

### Super-node
- Hardware: Raspberry Pi 4 or equivalent mini PC
- OS: Ubuntu Server or Raspberry Pi OS
- Backend: Node.js or Python (Flask/FastAPI)
- Database: SQLite (sufficient for v1 scale)
- Dashboard: Simple web app (React or plain HTML/JS) served locally
- Internet requirement: Only for M-Pesa verification and remote monitoring

### Development approach for v1
Build in this order:
1. Client app with internet-only broadcast (no mesh yet) — proves core UX
2. Super-node dashboard with approval queue and token system
3. Add WiFi Direct sync layer
4. Add Bluetooth sync layer
5. Add full bidirectional gossip sync

---

## 13. Key constraints (never violate these)

- Text and emoji only — no images, video, audio, or file attachments ever
- Countdown starts on RECEIPT by each node, never on send
- Tokens are single-use and tied to one NODE ID only
- Super-node must approve every post before it enters the network
- Free post: 1 per user per calendar month, 2-day duration only
- NODE ID is permanent and generated offline — never reassigned
- All sync is bidirectional — data flows both ways in every session
- Credits are always deducted at approval, never at submission
